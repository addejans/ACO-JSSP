\begin{abstract}
This report updates the original ACO-JSSP project documentation to reflect the modern Python 3 solver, generated instances, browser-based Gantt visualizer, HiGHS mixed-integer linear programming baseline, comparison runner, test suite, and GitHub Pages deployment. The project demonstrates how Ant Colony Optimization can be used as a constructive metaheuristic for the Job Shop Scheduling Problem and how a solver-backed MILP formulation can measure optimality gaps on smaller instances. The current implementation separates problem parameters from algorithm parameters, supports generated instances, applies a remaining-work-aware transition heuristic, mixes exploration with exploitation, reinforces both iteration-best and global-best solutions, applies pair-swap local search, and compares heuristic schedules against exact or bounded exact results.
\end{abstract}\vfill\end{titlepage}

\section{Introduction}
The Job Shop Scheduling Problem (JSSP) is a canonical problem in operations research and combinatorial optimization. A set of jobs must be processed on a set of machines. Each job has a fixed route through the machines, and each operation has a processing time. A feasible schedule must respect job precedence and machine capacity.

This repository began as a 2017 research prototype applying Ant Colony Optimization (ACO) to a small job-shop instance. The original version was useful as a learning artifact but difficult to maintain: it used Python 2 syntax, relied on an old Plotly API, used heavy global state, included hard-coded plotting credentials, and did not include tests or a modern UI. The updated repository keeps the historical scripts in \texttt{legacy/}, but moves active work into a cleaner Python 3 implementation with tests, a static UI, an exact MILP baseline, and report documentation.

\section{Problem Definition}
Let $J = \{1,\ldots,n\}$ be the set of jobs and $M = \{1,\ldots,m\}$ be the set of machines. Each job $j$ contains an ordered route of operations, $O_j=(O_{j,1},\ldots,O_{j,m})$. Each operation $O_{j,k}$ has processing time $p_{j,k}>0$ and required machine $\mu_{j,k}\in M$. In generated instances, each job visits each machine exactly once in a random route order.
\subsection{Job precedence} \[s_{j,k+1}\geq s_{j,k}+p_{j,k}\quad \forall j,\;k=1,\ldots,m-1.\]
\subsection{Machine capacity} For any two operations requiring the same machine, processing intervals cannot overlap: \[[s_{j,k},s_{j,k}+p_{j,k})\cap[s_{j',k'},s_{j',k'}+p_{j',k'})=\emptyset.\]
\subsection{Objective} The repository minimizes makespan: \[C_{\max}=\max_{j,k}(s_{j,k}+p_{j,k}).\]

\section{ACO Solver Design}
The ACO solver constructs a job-order sequence. If there are $n$ jobs and $m$ machines, the sequence has length $n\times m$. Each time job $j$ appears, the decoder schedules the next unscheduled operation of that job at the earliest feasible time.
\subsection{Schedule decoding} The decoder tracks job readiness, machine readiness, and the next operation index for each job. When job $j$ is selected, the next operation starts at \[s=\max(\text{jobReady}_j,\text{machineReady}_{\mu_{j,k}}).\]
\subsection{Transition rule} At each construction step, an ant chooses among jobs that still have unscheduled operations. A job's attractiveness depends on pheromone and a heuristic score: \[P(j)=\frac{\tau_{i,j}^{\alpha}\eta_j^{\beta}}{\sum_{\ell\in F}\tau_{i,\ell}^{\alpha}\eta_{\ell}^{\beta}}.\] The updated heuristic is remaining-work-aware: \[\eta_j=\frac{1}{\max(\text{earliestFinish}_j+0.15\cdot\text{remainingWork}_j,1)}.\]
\section{Additional ACO Techniques Implemented}
\subsection{Problem versus algorithm parameters} The UI separates problem parameters from algorithm parameters. Problem parameters define the instance: jobs, machines, duration range, and instance seed. Algorithm parameters define the search: ants, iterations, local search budget, and algorithm seed.
\subsection{Exploration and exploitation} The solver mixes probabilistic weighted choice with a small amount of greedy exploitation. With probability $q_0$, an ant chooses the highest-weight feasible move; otherwise it samples from the weighted transition distribution.
\subsection{Pheromone evaporation and elite reinforcement} After each iteration, pheromone evaporates: \[\tau_{i,j}\leftarrow(1-\rho)\tau_{i,j}.\] The solver reinforces the iteration-best order and applies an elite deposit to the global-best order: \[\tau_{i,j}\leftarrow\tau_{i,j}+\frac{Q}{C_{\max}}.\]
\subsection{Pair-swap local search} After an ant constructs a full job-order sequence, the solver applies pair-swap local search. It swaps two positions, decodes the candidate, and keeps the move if makespan does not worsen.
\section{Exact MILP Baseline with HiGHS}
The repository now includes an exact mixed-integer linear programming baseline in \texttt{src/milp\_jssp.py}. The baseline is not meant to replace ACO. It is a measuring stick. For small and moderate instances, HiGHS can prove optimality or return a solver status, objective value, runtime, and MIP gap.
\subsection{MILP variables} The formulation uses start-time variables $s_o$, makespan variable $C_{\max}$, and binary ordering variables $y_{a,b}$ for each pair of operations sharing a machine.
\subsection{Makespan constraints} For every operation $o$ with processing time $p_o$: \[C_{\max}\geq s_o+p_o.\]
\subsection{Disjunctive machine constraints} For two operations $a$ and $b$ on the same machine, let $y_{a,b}=1$ mean operation $a$ precedes operation $b$. With horizon $H$, the disjunction is linearized as: \[s_b-s_a-Hy_{a,b}\geq p_a-H,\] \[s_a-s_b+Hy_{a,b}\geq p_b.\]
\subsection{Comparison runner} After installing SciPy, ACO and HiGHS can be compared from the command line:
\begin{lstlisting}[language=bash]
python -m pip install -r requirements-optional.txt
python -m src.compare_solvers --jobs 4 --machines 3 --ants 40 --iterations 120 --milp-time-limit 10
\end{lstlisting}
The output includes ACO makespan, ACO runtime, HiGHS objective, HiGHS runtime, solver status, MIP gap, and ACO gap when HiGHS proves optimality.

\section{Limitations}
The HiGHS MILP baseline is valuable for small instances and quality comparisons, but exact formulations can become expensive as the number of jobs and machines grows. The browser exact search is deliberately capped to keep GitHub Pages responsive. The current ACO model learns pheromone on transitions between job selections rather than richer operation-level or machine-level sequencing decisions.
\section{Future Work}
\begin{enumerate}[leftmargin=0.25in]
\item Add standard benchmark instances such as FT06, LA, ORB, and ABZ families.
\item Extend the HiGHS comparison runner with benchmark tables and solver diagnostics.
\item Add a CP-SAT baseline for comparison against the MILP formulation.
\item Add critical-path local search neighborhoods.
\item Add machine-specific pheromone matrices.
\item Add dispatching-rule hybrids such as SPT, LPT, slack, and bottleneck-aware rules.
\item Add runtime and convergence charts.
\item Export schedules to CSV and JSON from the browser UI.
\end{enumerate}
\section{Conclusion}
The updated ACO-JSSP repository is now a coherent optimization demonstration. It includes a Python ACO solver, generated instances, an exact HiGHS MILP baseline, a solver comparison runner, a static browser UI, Gantt visualization, unit tests, GitHub Pages deployment, and a LaTeX-style technical report. The practical lesson is that metaheuristics become much more useful when paired with clean representations, validation, exact baselines, reproducibility, and visualization.
